> 基于 `origin_game/` 第一代版本（HTML 单文件版 + 多端联网版）的分析总结，为微信小程序版提供参考。

## 1. 棋盘模型

### 1.1 六边形网格

棋盘采用 **17 行六边形网格**，使用轴向坐标 (q, r) 进行数学运算。

```
行布局 ROWS_LAYOUT: [1, 2, 3, 4, 13, 12, 11, 10, 9, 10, 11, 12, 13, 4, 3, 2, 1]
总行数: 17 (索引 0-16)
最大列: 17
```

棋盘呈菱形/六边形轮廓——顶部和底部逐渐收窄为三角形（行 0-3 和行 13-16 是窄行），中间行（行 4-12）较宽。

### 1.2 坐标系统

- **网格坐标 (r, c)**：用于数组索引，`board[r][c]`
- **轴向坐标 (q, r)**：用于六边形方向计算的数学坐标
- 转换公式：`q = c - floor(r / 2)`, `axialR = r`
- 双向映射 `indexToAxial` 和 `axialToIndex` 支持快速查找

### 1.3 格子状态

| 值 | 常量 | 含义 |
|----|------|------|
| -1 | `INVALID` | 无效格（棋盘外区域） |
| 0 | `EMPTY` | 空格子 |
| 1 | `OBSTACLE` | 障碍物（棋子，可被吃掉） |
| 2 | `PLAYER` | 玩家当前所在位置 |

### 1.4 障碍物颜色

障碍物使用 5 种颜色区分（用于视觉美观，不影响规则）：

| 颜色名 | 色值 | 用途 |
|--------|------|------|
| PURPLE | `#A020F0` | 上层障碍物 |
| BLUE | `#3282FF` | 中上层障碍物 |
| PINK | `#FF69B4` | 中层障碍物 |
| YELLOW | `#FFD700` | 中下层障碍物 |
| ORANGE | `#FF8C00` | 下层障碍物 |
| RED | `#DC143C` | 底层障碍物 |

棋盘使用 `board` 数组存储格子类型，`boardColors` 数组并行存储颜色信息。

## 2. 游戏机制

### 2.1 核心规则：障碍跳跃

这是游戏的名称由来——**"障碍跳棋"**：

- 玩家棋子沿六边形网格的 6 个方向之一跳跃
- 跳跃距离必须是偶数：2、4、6 或 8 格
- **关键约束**：跳跃路径的中点必须恰好有一个障碍物——玩家"吃掉"它
- **路径清空约束**：跳跃路径上除中点外的所有格子必须为空
- 跳跃后：起点变空、中点障碍物被移除、终点变为玩家位置

```
跳跃示意（以因子 4 为例）：
  🟢 → ○ → 🔴 → ○ → ○
  起点  空  障碍  空  终点
          (被吃掉)
```

### 2.2 六方向移动

轴向坐标的 6 个方向向量：

```js
AXIAL_DIRS = [
    [1, 0], [1, -1], [0, -1],    // 右、右上、左上
    [-1, 0], [-1, 1], [0, 1]     // 左、左下、右下
]
```

### 2.3 胜利条件

玩家棋子到达**顶部三角形区域**（行索引 ≤ 3，即前 4 行），游戏胜利。

### 2.4 移动验证算法 (`canJumpTo`)

1. 检查起点和终点都在棋盘内
2. 起点必须是 PLAYER，终点必须是 EMPTY
3. 判断是否方向对齐（六方向之一）且距离为偶数（2/4/6/8）
4. 验证中点（距离的一半处）恰好是 OBSTACLE
5. 检查路径上所有中间格子（除中点外）都是 EMPTY

## 4. 用户交互设计

### 4.1 交互流程

```
启动页面 → 点击"开始" → 计时器启动 → 游戏进行中 →
→ 点击空格子尝试跳跃 → 验证是否合法 →
```

## 5. 计时与积分系统

### 5.1 双重计时器

| 计时器 | 方向 | 精度 | 说明 |
|--------|------|------|------|
| 游戏总时间 | 倒计时 | 100ms | 从 10:00 减到 0:00，归零触发结算 |
| 落子耗时 | 正计时 | 10ms | 每步重置为 0，显示秒.百分秒（如 15.23 秒） |

### 5.2 计时惩罚机制

```
基准阈值：15 秒
之后每 10 秒：额外扣 3 分
第 1 个区间 (15s-25s)：扣 3 分
第 2 个区间 (25s-35s)：扣 3 分（累计 6 分）
...
```

### 5.3 落子耗时视觉反馈

| 时间区间 | 视觉效果 |
|----------|---------|
| 0-10s | 默认颜色 |
| 10-15s | 消息栏显示 "快落子！要扣分啦！" |
| 15-30s | 颜色从黑色线性渐变到红色（与时间成正比） |
| 30s+ | 保持深红色 `rgb(100,0,0)` + 弹跳缩放动画 |

### 5.4 积分规则

| 事件 | 积分变化 |
|------|---------|
| 初始分值 | 150 分 |
| 悔棋 | -5 分/次 |
| 落子超时 | -3 分/区间（每 10 秒） |
| 最低分值 | 0 分（不出现负数） |

### 5.5 最终结算

```
factor1 (完成度因子)：
  - 通关：剩余时间 / 总时间（越早通关越高）
  - 未通：正确步数 / 目标步数

factor2 (积分因子)：当前积分 / 150

最终得分 = round((factor1 + factor2) × 50)  →  范围 0-100
```

## 6. 关卡生成系统

### 6.1 生成算法：逆向生长 (`growPathElegant`)

关卡生成采用**从终点反向生长**的策略：

1. 随机选择顶部三角区域（行 0-3）的一个位置作为终点
2. 把玩家棋子放在该位置，向下方寻找合法的逆向移动（即"反向跳跃"——把空格变成障碍物并跳回更远处）
3. 每一步对所有候选逆向移动进行**启发式评分**：
   - **垂直引导分**：鼓励向下移动（行号越大分数越高），进入收官阶段（>85%进度）强制向下
   - **回跳奖励**：如果移动方向上同时又向底部推进，给予额外加分
   - **距离分**：距原始终点越远，评分越高（鼓励探索）
   - **拥挤度惩罚**：周围障碍物太多时扣分（保证棋盘稀疏）
   - **高质量模式额外**：长跳奖励、横向展开奖励、更严格的 BFS 验证频率
4. 按分数排序，选择 top-N 候选继续递归
5. 每 N 步运行一次 `fastSolver` (BFS) 验证依然存在至少 `currentDepth` 步的解法
6. 到达目标步数后，进行**唯一解校验**

### 6.2 唯一解校验 (`verifyUniqueSolution`)

1. BFS 求出主解路径
2. 遍历路径的每一步，检查是否有**分叉**（同一位置可以跳到另一个合法位置）
3. 对每个分叉，尝试移除对应障碍物后用 BFS 求解——如果能到达终点，说明解不唯一
4. 全部通过则为唯一解

### 6.3 生成参数

| 参数 | 说明 |
|------|------|
| 目标步数 | 5/10/15/20/25（从下拉框选择或自定义） |
| 强制底部起点 | 确保玩家起始位置在底部 4 行内 |
| 高质量模式 | 增加候选宽度、更严格的步数验证、长跳和横向展开优化 |
| 超时 | 基础 15s，目标 >20 步时为 45s |

### 6.4 关卡数据格式

```json
{
    "obstacles": [[row, col, "#hexcolor"], ...],
    "player": [row, col],
    "solutionPath": [[row, col], ...]  // BFS求解的完整路径
}
```

关卡可以 Base64 编码后导出/导入。

```
CELL_SIZE = 33px
```

## 9. 多端联网版架构（参考价值）

虽然目标是小程序单机版，但多端版有一些值得了解的设计：

### 9.1 项目结构

```
origin_game/
├── 障碍跳棋_新.html    # 单文件完整版（Canvas + 生成 + 音效 + 计时 + 结算）
├── 障碍跳棋.html       # 旧版
├── countdown.html       # 倒计时独立组件
├── server.js            # Express + Socket.IO 入口
├── server/
│   ├── GameState.js     # 权威游戏状态（撤销栈、计时器、分数、序列化）
│   └── SocketHandler.js # Socket.IO 事件路由和房间管理
├── client/
│   ├── login.html       # 登录页（输入显示名称）
│   ├── player.html      # 玩家游戏界面（Canvas + 控件）
│   ├── player.js        # Canvas 渲染、Socket 客户端
│   ├── background.html  # 管理面板（生成/导出/导入关卡、查看日志）
│   ├── background.js    # 管理面板逻辑
│   ├── showup.html      # 观战模式（两个棋盘并排展示两队的实时状态）
│   ├── showup.js        # 观战 Canvas 渲染
│   └── style.css        # 公共样式
└── shared/
    ├── constants.js     # UMD 模块：棋盘布局、方向向量、颜色、计时常量
    └── game-logic.js    # UMD 模块：棋盘操作、移动验证、BFS 求解、关卡生成
```

### 9.2 关键设计决策

- **无构建工具**：shared/ 使用 UMD 包装器，同时支持 Node.js `require()` 和浏览器 `<script>` 全局引用
- **服务端权威**：所有游戏状态变更由服务端 `GameState` 执行，客户端 `GameLogic` 仅用于渲染和可选移动预览
- **双独立状态**：两支队伍各有独立的棋盘、计时器和分数，永不共享
- **Socket.IO 房间**：`teamA`、`teamB`、`observer`，事件名 `game:state` / `observer:state`

## 10. 微信小程序版迁移要点

### 10.1 可直接复用的逻辑

- 棋盘模型（`ROWS_LAYOUT`、`INVALID/EMPTY/OBSTACLE/PLAYER`）
- 坐标映射（indexToAxial / axialToIndex）
- 移动验证算法（`canJumpTo`）
- BFS 求解器（`solveGameBFS`）
- 关卡生成算法（`growPathElegant` + `verifyUniqueSolution`）
- 逆向移动逻辑（`getAllReverseMoves`）
- 积分与计时规则常量

### 10.2 需要替换的部分

| HTML 原版 | 小程序替代方案 |
|-----------|---------------|
| `<canvas>` + 2D Context API | `<canvas>` + uni-app Canvas API（或有 Canvas 2D 模式） |
| `addEventListener('click')` | `@tap` 或 `bindtap` |
| `addEventListener('mousemove')` | `@touchmove`（触控悬停在小程序中实现方式不同） |
| `mousemove` 箭头预览 | 可考虑点击选中目标后直接绘制箭头，或省略 |
| `setInterval` 计时器 | `setInterval`（小程序支持） |
| Web Audio API 音效 | `uni.createInnerAudioContext()` 或 `wx.createInnerAudioContext()` |
| `navigator.clipboard.writeText()` | `uni.setClipboardData()` |
| `alert()` | `uni.showToast()` / `uni.showModal()` |
| `document.getElementById()` | `uni.createSelectorQuery()` 或数据绑定 |
| CSS 动画 / `@keyframes` | WXSS（小程序 CSS 子集）或 `uni.createAnimation()` |
| `localStorage` | `uni.setStorageSync()` / `uni.getStorageSync()` |

### 10.3 需要重新设计的交互

1. **移动操作**：HTML 版直接点击目标位置。小程序可考虑：
   - 直接点击目标格子（最简单，保持原体验）
   - 滑动方向选择（更符合触控习惯）
   
2. **悬停预览**：小程序没有 `mousemove`，可改为：
   - 点击玩家棋子显示所有可跳目标
   - 或省略箭头预览，只保留高亮目标环

3. **控制面板**：HTML 版的折叠式面板在小程序中可改用 `uni.showActionSheet()` 或底部抽屉

4. **结算面板**：HTML 版动态创建 DOM，小程序中应作为组件或条件渲染

5. **Canvas 适配**：`750×600` 的固定画布需要在小程序中响应式适配不同屏幕
