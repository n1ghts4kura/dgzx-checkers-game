# uni-app 组成和跨端原理

## 基本语言和开发规范

uni-app 的代码编写以 JS、Vue、CSS 为基础语言，同时支持 TypeScript 及 SCSS 等 CSS 预处理器。

在 App 端，还额外支持原生渲染的 **nvue**，以及可编译为 Kotlin 和 Swift 的 **uts** 语法。

DCloud 同时提供了 uniCloud 云引擎，让开发者能用 JS 编写服务端代码。这意味着只要掌握 JS，就能覆盖 Web、Android、iOS、各家小程序及服务端的全栈开发。

为保证多端兼容，综合考虑编译速度与运行性能等因素，uni-app 约定了以下开发规范：

- 页面文件遵循 **Vue 单文件组件 (SFC) 规范**，每个页面为一个 `.vue` 文件
- 组件标签接近小程序规范，详见 [uni-app 组件规范](/component/)
- JS API 接近小程序规范，但需将 `wx`、`my` 等前缀替换为 `uni`，详见 [uni-app 接口规范](/api/)
- 数据绑定与事件处理遵循 **Vue.js** 规范，同时补充了应用生命周期及页面生命周期
- 如需兼容 app-nvue 平台，建议使用 flex 布局

## 编译器与运行时

uni-app 由**编译器**和**运行时（runtime）**两部分组成。实现一套代码、多端运行的能力，正是通过这两者的协同配合来完成的。

### 编译器

- 编译器运行在电脑开发环境，通常内置于 HBuilderX 中，也可使用独立 CLI 版
- 开发者按 uni-app 规范编写代码，编译器将其编译为各平台支持的特有代码：
  - **Web 平台**：将 `.vue` 文件编译为 JS 代码，与普通 Vue CLI 项目类似
  - **微信小程序平台**：将 `.vue` 文件拆分为 wxml、wxss、js 等代码
  - **App 平台**：将 `.vue` 文件编译为 JS 代码。涉及 uts 时：
    - Android 平台：将 `.uts` 文件编译为 Kotlin 代码
    - iOS 平台：将 `.uts` 文件编译为 Swift 代码
- 编译器分为 Vue2 版和 Vue3 版：
  - Vue2 版：基于 **Webpack** 实现
  - Vue3 版：基于 **Vite** 实现，性能更优
- 编译器支持**条件编译**，可指定某些代码只编译到特定终端平台

```javascript
// #ifdef  App
console.log("这段代码只有在App平台才会被编译进去。非App平台编译后没有这段代码")
// #endif
```

### 运行时（runtime）

Runtime 不在开发环境中运行，而是运行在真正的终端设备上。uni-app 在每个平台都有各自独立的 runtime。

- **在小程序端**：uni-app 的 runtime 主要是一个适配小程序的 Vue runtime，页面路由、组件及 API 方面基本是转译映射
- **在 Web 端**：相比普通 Vue 项目，多了一套 UI 库、页面路由框架和 `uni` 对象
- **在 App 端**：uni-app 的 runtime 更为复杂，可简单理解为 DCloud 将开发者的代码与自研小程序引擎打包为 apk 或 ipa

uni-app runtime 包含三部分：**基础框架**、**组件**、**API**。

#### 基础框架

- 包括语法、数据驱动、全局文件、应用管理、页面管理、JS 引擎、渲染与排版引擎等
- Web 和小程序上直接使用浏览器或小程序自身的 JS 引擎与排版引擎，App 端则需要 uni-app 提供
- App 的 JS 引擎：Android 上为 V8，iOS 上为 JavaScriptCore
- App 的渲染引擎：同时提供两套——`.vue` 页面由 WebView 渲染，`.nvue` 页面由原生渲染

#### 组件

- Runtime 中仅包含基础组件，如 `<view>`、`<button>` 等；扩展组件需下载到项目代码中
- 内置基础组件的命名规范与小程序基本一致
- 在小程序端，基础组件直接转译为小程序自身的内置组件
- 在 Web 和 App 端，这些组件位于 uni-app runtime 中，相当于内置了一套 UI 库
- 大多数扩展组件都基于基础组件封装，如官方扩展 UI 库 **uni ui**
- App 平台支持用原生语言扩展原生组件

#### API

- uni-app runtime 内置了大量跨端 API，如 `uni.request`（联网）、`uni.getStorage`（读取存储）
- 小程序平台：`uni` 对象会转为小程序自有对象，如微信小程序中 `uni.request` 等同于 `wx.request`
- Web 平台：`window`、`dom` 等浏览器专用 API 仍可使用
- App 平台：除 `uni.` API 外，还可使用 `plus.` API、Native.js，以及通过 uts、Java 或 Objective-C 编写的原生插件
- **ext API**：部分不常用的 API 被剥离为 ext API，虽仍以 `uni.` 开头，但需单独下载插件

## 逻辑层和渲染层分离

在 Web 平台，逻辑层（JS）与渲染层（HTML、CSS）运行在同一个 WebView 中。

但在小程序和 App 端，逻辑层与渲染层被分离了。分离的核心原因在于**性能**——过去基于 WebView 的 App 性能不佳，很大程度上是因为 JS 运算与界面渲染相互抢占资源导致的卡顿。

无论小程序还是 App，逻辑层都已独立为单独的 JS 引擎，渲染层仍然是 WebView（App 端也支持纯原生渲染）。

**注意**：小程序和 App 的逻辑层**不支持**浏览器专用的 `window`、`dom` 等 API。App 只能在渲染层操作 window 和 dom，即通过 renderjs 实现。
