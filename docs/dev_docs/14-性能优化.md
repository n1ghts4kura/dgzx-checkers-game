# 性能优化专题

## 运行原理

### 逻辑层和视图层分离

`uni-app` 在非 H5 端运行时，架构上分为逻辑层和视图层两部分。逻辑层负责执行业务逻辑，视图层负责页面渲染。

**逻辑层**运行在独立的 `jscore` 中，不依赖于本机的 webview。优点是没有浏览器兼容问题；缺点是无法使用 `window`、`document`、`navigator`、`localstorage` 等浏览器专用 JS API。

| 平台 | 视图层实现 |
|------|-----------|
| H5、小程序、app-vue | webview |
| app-nvue | 基于 weex 改造的原生渲染视图 |

### 分离的利与弊

**优势：**
- JS 运算不卡渲染，窗体动画稳定
- 无需书写预载代码，新窗体渲染快且动画稳定

**劣势：**
- 两层通信存在损耗。iOS 影响较小，但 Android 低端机上每次通信耗时几十毫秒

**解决方案：**
- **webview 渲染时**：使用 wxs 或 renderjs 运行于视图层
- **原生渲染时**：使用 bindingx 机制

### app-vue 和小程序的数据更新

对于复杂页面，需要将某区域更新做成组件，否则更新该区域数据时会触发整个页面的数据同步，造成点击延迟卡顿。例如微博长列表：点击点赞图标时，点赞按钮必须做成组件。

> app-nvue 和 H5 不存在此问题。

## 优化建议

### 避免使用大图

大量使用大图资源会造成页面切换卡顿、系统内存升高，甚至白屏崩溃。不要将多张大图缩小后显示在一个屏幕内。

### 优化数据更新

data 中的数据每次变化都会通知视图层重新渲染。**非视图所需的变量不要定义在 data 中**，可在外部定义或直接挂载在 vue 实例上。

### 长列表

1. **点赞按钮需独立为组件**：实现差量数据更新
2. **合理引入组件**：如无差量更新需求，不需要将每个 item 做成组件
3. **数据分页**：单个组件存在大量数据时，建议用组件对数据分页
4. **App 端 nvue 长列表**：应使用 list 组件（自动渲染资源回收）。推荐使用 uni-ui 的 uList 组件
5. **左右滑动长列表**：参考新闻模板的标杆实现

### 减少一次性渲染的节点数量

建议局部更新页面，如服务端返回 100 条数据可分批加载。

### 减少组件数量、减少节点嵌套层级

深层嵌套节点在页面初始化时占用更多内存，遍历节点也更慢。部分 nvue 页面在 Android 低端机上初次渲染时出现从上到下的渲染过程，多因组件过多。

### 避免视图层和逻辑层频繁通讯

- 减少 scroll-view 的 scroll 事件监听
- 监听滚动时不要实时改变 scroll-top/scroll-left
- 注意 onPageScroll 的使用
- 多用 CSS 动画，而非 JS 定时器操作界面做动画
- canvas 跟手操作：App 端用 renderjs，小程序端用 web-view 组件

### 优化页面切换动画

1. **延时渲染**：页面初始化时延时 100ms~300ms 渲染图片或复杂组件
2. **选择合适的动画效果**：popin/popout 资源消耗更大，可使用 slide-in-right/slide-out-right
3. **页面预载**：App-nvue 和 H5 支持 uni.preloadPage

### 优化背景色闪白

- 将样式写在 App.vue 中，加速页面样式渲染速度
- App 端可在 pages.json 的页面 style 中配置原生背景色：

```json
"style": {
    "app-plus": {
        "background":"#000000"
    }
}
```

### 优化启动速度

1. 控制工程体积：背景图和本地字体文件越大，对小程序启动速度影响越大
2. fast 启动模式：首页为 nvue 页面并设置为 fast 启动模式时启动速度最快
3. 纯 nvue 项目：manifest 中设置 app-plus → renderer: "native"，约 2 秒完成启动
4. 分包：App 也支持分包，启动时只加载部分 JS

### 优化包体积

**小程序端**：自带引擎只有几十 K（定制过的 vue.js 核心库）

**H5 端**：提供摇树优化机制，未优化前约 500k（gzip 后 162k），需在 manifest 中配置开启

**App 端**：
- Android 基础引擎约 9M
- 提供扩展模块（地图、蓝牙等），打包时不需要的模块可裁剪
- 纯 nvue 项目可再减少约 2M 体积
- HBuilderX 2.7 起默认不再包含 x86 CPU 支持
